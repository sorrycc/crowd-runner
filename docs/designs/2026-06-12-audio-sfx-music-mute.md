# Swarm Run — Audio: SFX + Looping Music + Mute Toggle

> **Revised during implementation:** (1) `unlock()` now calls `ctx.resume()`
> *synchronously* (un-awaited) immediately after creating the context, so the resume stays
> inside the Start-click gesture stack — most robust for autoplay policy (esp. Safari);
> subsequent restart calls use the awaited resume (sticky activation). (2) Added
> `build.assetsInlineLimit: 0` to `vite.config.js` so all 13 clips emit as fingerprinted
> files instead of small ones inlining as base64 in the JS bundle (matches "Vite
> fingerprints them"). No behavior/AC change.

## 1. Background

Swarm Run is a Three.js + Vite crowd-shooter that is silent by design (README: "no
external assets, no audio"). This feature introduces the first audio: a comprehensive
set of event SFX, a looping background-music bed, and a HUD mute toggle. It intentionally
relaxes the "no external assets" rule for audio only — bundled audio **files** (not
runtime-procedural WebAudio synthesis), CC0/royalty-free, `.mp3`, with provenance
documented in the repo.

## 2. Requirements Summary

**Goal:** Add bundled-file audio — 12 event SFX + 1 looping music bed + a persisted HUD
mute toggle — to the currently-silent game, without touching balance/verify logic.

**In scope:** `AudioManager` (`src/core/Audio.js`) using WebAudio (decoded buffers +
shared cached `AudioBuffer`s played through short-lived per-shot `BufferSource` nodes +
master `GainNode`); real, playable CC0 `.mp3` assets under
`src/assets/audio/` generated by a committed reproducible `ffmpeg` script; a `CREDITS.md`
file; constructor-injection into `Game`; SFX fired from existing `Game.js` hooks; a mute
button in `HUD.js`.

**Out of scope:** runtime procedural synthesis; verifier smoke assertion for audio
(YAGNI); visibility/focus resume listeners (YAGNI); third-party (freesound/Kenney) asset
procurement (self-authored instead); volume-mix UI; multiple tracks; spatial audio.

**Key decisions:** see Decision Log (section 5).

## 3. Acceptance Criteria

1. Real, playable CC0 `.mp3` files committed under `src/assets/audio/`, generated by a
   committed reproducible `ffmpeg` script, with provenance/license/generator-command
   documented in a `CREDITS.md` file.
2. `AudioManager` (`src/core/Audio.js`) loads/decodes each sound once into a cached,
   reused `AudioBuffer`, and plays it by spawning a fresh short-lived `BufferSource` per
   call routed through a master `GainNode` — so concurrent/overlapping SFX (e.g. gunshots)
   never cut each other off. (The issue's "pooled BufferSource" wording maps to this
   standard WebAudio pattern: `AudioBufferSourceNode`s are single-use by spec, so the
   *buffer* is the reused/cached resource, not the source node.) Supports live mute via the
   master gain and enforces a ~60ms per-sound min-interval guard.
3. Every mapped event triggers its sound from the real `Game.js` hooks: weapon fire
   (`_fire`, throttled to a separate ~0.11s SFX cadence), gate good/bad
   (`_resolveCrossings`), power-up pickup (`_applyPowerup`), block destroyed (Obstacle
   hp→0 edge), enemy squad down (Enemy hp→0 edge), soldier hurt (contact drain /
   boss-bullet hit), boss shot fired (`boss.update` telegraph), boss defeated (boss hp≤0),
   stage advance (`_advanceStage`/banner flash), win and lose stings (`_end`).
4. Music loops during PLAYING, continues seamlessly across stage advance, stops on
   WIN/LOSE with a one-shot win/lose sting, and restarts from the top on restart.
5. HUD mute toggle defaults unmuted, persists to `localStorage` (try/catch, silent no-op
   if unavailable, no console errors), and silences SFX + music immediately when toggled.
6. Audio unlocked by the Start gesture (best-effort `ctx.resume()` on Start/Restart);
   nothing plays before the first gesture; no autoplay-blocked console errors.
7. No gameplay or perf regression; `npm run verify` still passes and remains fully
   decoupled from audio (Game runs correctly with a null/absent AudioManager via guarded
   call sites).
8. README "no audio" claim updated to reflect the new audio system.

## 4. Problem Analysis

- **HTMLAudioElement per SFX** — simple, but single-playback (overlapping gunshots cut
  each other off) and high latency. Rejected: fails AC2's overlapping-SFX intent.
- **Runtime procedural WebAudio (oscillators at play time)** — no asset files, but the
  triage explicitly chose bundled files; harder to make a rich music bed. Rejected by
  triage decision.
- **Chosen: WebAudio cached decoded `AudioBuffer`s + fresh per-shot `BufferSource` +
  master gain** — concurrent SFX, low latency, single master gain for mute/volume. Matches
  AC2's intent exactly. Assets are
  pre-rendered `.mp3` files imported through Vite (fingerprinted), decoded once on unlock.

## 5. Decision Log

**1. Asset sourcing (the one external dependency)**
- Options: A) source from freesound/Kenney/OpenGameArt · B) self-author CC0 via a
  committed `ffmpeg` generator script · C) commit silent stubs + a sourcing manifest
- Decision: **B)** — this environment can't reliably fetch binary assets from those sites;
  self-authored audio is unambiguously CC0/public-domain (no attribution burden), produces
  real playable files (not stubs, so AC1/AC3 are genuinely met), and keeps the build
  self-contained. CREDITS documents provenance + the exact generator command and notes the
  files are swappable for library assets later. Surfaced to the user; proceeding on the
  recommended default (reversible — files can be swapped without code change).

**2. Asset import strategy**
- Options: A) one explicit `import url from '...'` per file · B) `import.meta.glob` over
  the folder
- Decision: **B)** — `import.meta.glob('../assets/audio/*.mp3', { eager: true, query:
  '?url', import: 'default' })`. DRY: one statement fingerprints every file; adding a sound
  is just dropping a file. The filename minus `.mp3` is the sound key.

**3. AudioContext creation timing**
- Options: A) construct context eagerly · B) construct on first `unlock()` gesture
- Decision: **B)** — fetch raw `ArrayBuffer`s eagerly (no context required), construct the
  `AudioContext` + decode buffers only inside `unlock()` (fired by the Start click). No
  context exists before a gesture → no autoplay-policy console warning (AC6).

**4. Boss-shot hook without coupling Boss to audio**
- Options: A) pass AudioManager into `Boss.update` · B) `Boss.update` returns a boolean
  "fired this frame"; Game plays the sound · C) diff the boss-bullet pool active count
- Decision: **B)** — `update()` returns `true` on the frame it fires (it already knows, via
  the internal `_fire` path). Pure fact, no audio coupling, Boss stays out of the verify
  path. Game: `if (this.track.boss.update(...)) audio?.play('boss-shot')`. Verify safety:
  `scripts/verify-balance.mjs` re-implements boss math inline from the stage configs (lines
  110-132) and imports **only** `STAGE_1`/`STAGE_2` — it never imports `Boss.js`, `Game.js`,
  or `Audio.js`, so this signature change and every `audio?.` hook are structurally outside
  the verify path. That import boundary — not "it still passes" — is the real AC7 guarantee.

**5. Block-break / enemy-down edge detection**
- Options: A) entities return a "just died" flag from `damage()` · B) Game checks
  `target.broken` / `target.dead` after calling `target.damage()`
- Decision: **B)** — the acquired target was live (not broken/dead) when selected, so a
  post-`damage()` check of `target.broken` (Obstacle) / `target.dead` (Enemy) captures the
  exact death frame. Zero entity changes; keeps the edge logic in Game where the audio
  lives.

**6. Mute mechanism + persistence**
- Options: A) master gain 0/1 · B) per-source disconnect · C) skip-play flag only
- Decision: **A + skip-play** — master `GainNode` ramped to 0/1 over ~15ms (click-free) is
  the truth and silences live music instantly; `play()` also early-returns when muted to
  avoid needless node allocation. State persisted to `localStorage['swarmrun.muted']`,
  read/written in try/catch (silent no-op if storage unavailable). Default unmuted.

**7. Weapon-fire SFX throttle**
- Options: A) reuse the 0.05s visual volley cadence · B) a separate, slower SFX cadence
- Decision: **B)** — `SHOOT_SFX_CADENCE = 0.11s` accumulator inside `_fire()`, independent
  of the 20/s visual volley rate, so gunfire reads as rapid fire without machine-gunning
  the audio. Backed by AudioManager's ~60ms global per-sound guard.

**8. Music lifecycle**
- Options: A) stop on WIN/LOSE · B) duck on WIN/LOSE
- Decision: **A)** — music **stops** on WIN/LOSE (clean end state) and a one-shot win/lose
  **sting** plays on the transition; music **continues seamlessly** across `_advanceStage`
  (no restart — still PLAYING); **restarts from the top** on `restart()`→`start()`.
  `playMusic()` is idempotent (won't stack sources). To make "restart from top"
  unconditional rather than relying on `_end` having already stopped music, `restart()`
  calls `audio?.stopMusic()` before delegating to `start()` (one line; removes the hidden
  coupling the reviewer flagged — restart is always reached from a WIN/LOSE screen today,
  but a future mid-PLAYING restart would otherwise silently no-op).

## 6. Design

### 6.1 AudioManager (`src/core/Audio.js`)

State: `ctx` (created on unlock), `master` GainNode, `buffers: Map<name,AudioBuffer>`,
`pending: Map<name,Promise<ArrayBuffer>>`, `muted` (from localStorage), `musicSource`,
`musicGain`, `_wantMusic` flag, `_lastPlay: Map<name,number>` for the min-interval guard,
`unlocked` flag.

Construction (no context yet):

```js
const MUTE_KEY = 'swarmrun.muted'
const MIN_INTERVAL = 0.06        // s, per-sound machine-gun guard
const MUSIC_VOL = 0.5
const RAMP = 0.015               // s, click-free mute ramp

export class AudioManager {
  constructor() {
    this.ctx = null; this.master = null; this.musicGain = null; this.musicSource = null
    this.buffers = new Map(); this.pending = new Map(); this._last = new Map()
    this._wantMusic = false; this.unlocked = false
    this.muted = this._loadMuted()
    this._fetchAll()             // eager fetch of ArrayBuffers (no AudioContext)
  }

  _fetchAll() {
    const mods = import.meta.glob('../assets/audio/*.mp3',
      { eager: true, query: '?url', import: 'default' })
    for (const [path, url] of Object.entries(mods)) {
      const name = path.split('/').pop().replace(/\.mp3$/, '')
      this.pending.set(name, fetch(url).then((r) => r.arrayBuffer()).catch(() => null))
    }
  }

  async unlock() {                       // called by the Start gesture
    if (!this.ctx) {
      const Ctx = window.AudioContext || window.webkitAudioContext
      if (!Ctx) return
      this.ctx = new Ctx()
      this.master = this.ctx.createGain()
      this.master.gain.value = this.muted ? 0 : 1
      this.master.connect(this.ctx.destination)
      if (this.ctx.state === 'suspended') this.ctx.resume()   // sync, in-gesture (AC6)
      await this._decodeAll()
      this.unlocked = true
      if (this._wantMusic) this.playMusic()
      return
    }
    // subsequent calls (restart): sticky activation → plain resume is fine
    if (this.ctx.state === 'suspended') { try { await this.ctx.resume() } catch {} }
  }

  async _decodeAll() {
    await Promise.all([...this.pending].map(async ([name, p]) => {
      const buf = await p; if (!buf) return
      try { this.buffers.set(name, await this.ctx.decodeAudioData(buf.slice(0))) } catch {}
    }))
  }

  play(name, { volume = 1, rate = 1 } = {}) {
    if (!this.ctx || this.muted) return
    const buf = this.buffers.get(name); if (!buf) return
    const now = this.ctx.currentTime
    if (now - (this._last.get(name) || -1) < MIN_INTERVAL) return    // spam guard
    this._last.set(name, now)
    const src = this.ctx.createBufferSource(); src.buffer = buf; src.playbackRate.value = rate
    const g = this.ctx.createGain(); g.gain.value = volume
    src.connect(g).connect(this.master); src.start()
  }

  playMusic() {
    if (!this.ctx) { this._wantMusic = true; return }    // queue until unlock
    if (this.musicSource) return                         // idempotent
    const buf = this.buffers.get('music-loop'); if (!buf) { this._wantMusic = true; return }
    this.musicGain = this.ctx.createGain(); this.musicGain.gain.value = MUSIC_VOL
    this.musicGain.connect(this.master)
    const src = this.ctx.createBufferSource(); src.buffer = buf; src.loop = true
    src.connect(this.musicGain); src.start()
    this.musicSource = src; this._wantMusic = false
  }

  stopMusic() {
    this._wantMusic = false
    if (this.musicSource) { try { this.musicSource.stop() } catch {}; this.musicSource = null }
  }

  setMuted(muted) {
    this.muted = muted; this._saveMuted(muted)
    if (this.master && this.ctx) {
      const t = this.ctx.currentTime
      this.master.gain.cancelScheduledValues(t)
      this.master.gain.setValueAtTime(this.master.gain.value, t)
      this.master.gain.linearRampToValueAtTime(muted ? 0 : 1, t + RAMP)
    }
  }
  isMuted() { return this.muted }

  _loadMuted() { try { return localStorage.getItem(MUTE_KEY) === '1' } catch { return false } }
  _saveMuted(m) { try { localStorage.setItem(MUTE_KEY, m ? '1' : '0') } catch {} }
}
```

Note: `decodeAudioData` consumes the ArrayBuffer; `.slice(0)` passes a copy so a future
re-decode can't fail on a detached buffer.

**Music start-up regimes (why `start()` calls both `unlock()` and `playMusic()`).**
`unlock()` runs synchronously up to its first `await` — `this.ctx`/`this.master` are
assigned before it yields — then awaits `_decodeAll()`. Two regimes:
- **First run:** `start()` calls `unlock()` (async, not awaited) then `playMusic()`
  synchronously. At that point `ctx` is set but `music-loop` is not decoded yet →
  `playMusic()` sets `_wantMusic = true` and returns. When `unlock()`'s `await
  _decodeAll()` resolves, its `if (this._wantMusic) this.playMusic()` actually starts the
  loop. So the post-decode call is load-bearing on first run.
- **Restart / subsequent start:** `ctx` + buffers already exist, so `unlock()` skips the
  `if (!this.ctx)` block (only re-`resume()`s) and the synchronous `playMusic()` in
  `start()` starts the loop immediately (buffer present). So the synchronous call is
  load-bearing on restart.
Both call sites are intentional; each covers one regime. `playMusic()`'s `if
(this.musicSource) return` keeps it idempotent so the two paths never stack a second loop.

### 6.2 Game wiring (`src/Game.js`)

- Constructor: `constructor(stages, audio = null)`; store `this.audio = audio`. Pass HUD
  the audio ref: `new HUD(audio)`.
- `start()`: `this.audio?.unlock(); this.audio?.playMusic()` (unlock is async/best-effort;
  playMusic queues until decode finishes — see the regime note in 6.1). Because `start()`
  runs inside the `btn-start` click handler stack (`Screens.onStart → Game.start`), the
  user gesture is preserved for `ctx.resume()` (AC6).
- `restart()`: add `this.audio?.stopMusic()` then delegate to `start()` (which re-runs
  `unlock()` → harmless: it skips context creation via `if (!this.ctx)` and only
  re-`resume()`s, and re-`playMusic()` from the top). Do **not** add a duplicate `unlock()`
  line — it already funnels through `start()`. The explicit `stopMusic()` makes
  "restart from the top" hold unconditionally (Decision 8).
- New constant `SHOOT_SFX_CADENCE = 0.11`, field `this._shootSfxAcc = 0` (reset in
  `_resetStageState`).
- `_fire(dt, ...)`: after emitting volleys, accumulate `this._shootSfxAcc += dt`; when it
  passes the cadence, `this.audio?.play('shoot', { volume: 0.5 })` and subtract the cadence
  (single play per cadence even if multiple frames elapsed).
- RUN combat: after `target.damage(F*dt)`, `if (target.broken) audio?.play('block-break')`
  `else if (target.dead) audio?.play('enemy-down')`. The acquired `target` is exactly one
  type — an **Obstacle** (exposes `.broken`, no `.dead`) or an **Enemy** (exposes `.dead`,
  no `.broken`); the two properties are mutually exclusive, so the `else if` reads the
  *absent* property as `undefined` (falsy) and selects correctly. This fires `enemy-down`
  ONLY from the focus-fire death edge — deliberately **not** from the "dodged / slipped
  past" path in `_resolveCrossings` (which also sets `e.dead = true` but is a dodge, not a
  kill, so it should be silent).
- `_resolveCrossings`: on gate apply, `audio?.play(good ? 'gate-good' : 'gate-bad')`; on
  block/enemy contact `drained > 0`, `audio?.play('hurt')`.
- `_applyPowerup`: `audio?.play('powerup')`.
- BOSS combat: `if (this.track.boss.update(...)) audio?.play('boss-shot', {volume:0.6})`.
- `_resolveBossBullets`: on an unshielded hit, `audio?.play('hurt')`. A multi-bullet frame
  is intentionally collapsed to one "ow" by AudioManager's 60ms per-name guard (one hurt
  per 60ms, not a dropped event).
- Win check: when `boss.hp <= 0` and time left, `audio?.play('boss-down')` once, then
  branch (advance vs WIN). The "once" is guaranteed by the control flow, not a latch:
  in the **WIN** branch `_end` sets `state='WIN'` so the next frame returns at the
  `state !== 'PLAYING'` guard; in the **advance** branch `_advanceStage`→`track.reset`
  restores full boss HP so `boss.hp <= 0` is false next frame. The 60ms guard is only a
  backstop. `_advanceStage` also plays `stage-advance`; `boss-down` + `stage-advance`
  intentionally **layer** on the advance frame (distinct names → the per-name guard does
  not dedupe them; kill sting + advance fanfare is desired).
- `_end(result)`: `audio?.stopMusic(); audio?.play(result === 'WIN' ? 'win' : 'lose')`.

All call sites use optional chaining so a null audio (or pre-decode) is a no-op (AC7).

### 6.3 Boss signal (`src/entities/Boss.js`)

`update()` returns `true` only on a frame it fired:

```js
update(dt, firepower, armyX, armyZ, bossBullets) {
  this.hp -= firepower * dt
  let fired = false
  if (this.hp > 0) {
    this._fireTimer += dt
    if (this._fireTimer >= this.fireInterval) {
      this._fireTimer -= this.fireInterval
      this._fire(armyX, armyZ, bossBullets); fired = true
    }
  }
  /* ...flash + hp tag unchanged... */
  return fired
}
```

### 6.4 HUD mute button (`src/ui/HUD.js` + `index.html`)

- `index.html`: a `#btn-mute` button in the HUD top-right corner, `pointer-events:auto`
  (HUD root is `pointer-events:none`), styled as a pill matching existing HUD chips.
- `HUD.js`: `constructor(audio)` stores `audio`, grabs `#btn-mute`, wires `click` →
  `audio.setMuted(!audio.isMuted()); this._renderMute()`. `_renderMute()` sets the icon
  (🔊 unmuted / 🔇 muted) and an `is-muted` class. Initial render reflects persisted state
  so a reload shows the right icon. The button is always visible while the HUD shows.
  **Scope (intentional):** the button lives inside `#hud`, which is hidden until PLAYING,
  so muting is **in-game only** — this matches the issue's "mute button in the HUD corner"
  wording. AC5 requires default-unmuted + persistence + live silence, none of which need a
  pre-Start control: a returning muted user's state is already applied (master gain inits
  from the persisted `muted` flag in `unlock()`, so the win/lose sting and music respect it
  even though the button isn't shown on the menu). No always-on corner button is added
  (YAGNI / follows the issue literally).

### 6.5 main.js injection


### 6.6 Assets + generator (`scripts/gen-audio.sh`)

A committed bash script runs `ffmpeg` to synthesize each file into
`src/assets/audio/`. Each SFX is a short (<0.6s) synth (sine/square/noise/sweep), encoded
mono `.mp3`; `music-loop.mp3` is a ~16s seamless loop (low-bitrate, small). The script is
idempotent and documented in CREDITS. Generated files: `shoot`, `gate-good`, `gate-bad`,
`powerup`, `block-break`, `enemy-down`, `hurt`, `boss-shot`, `boss-down`, `stage-advance`,
`win`, `lose`, `music-loop`.

`CREDITS.md` (repo root): documents that all audio is self-authored CC0 / public domain,
the generator command, and a note that files can be swapped for library assets without
code changes.

**Build / CI boundary.** The 13 `.mp3` files are **committed** (listed in section 7), and
the Vite dev/build and the GitHub-Pages deploy consume those committed files directly.
`scripts/gen-audio.sh` is a **dev-only** reproducibility tool that requires a local
`ffmpeg`; it is never invoked by `npm run build`, `npm run verify`, or CI. So no build/CI
path depends on `ffmpeg` being installed.

**Startup cost (AC7).** `AudioManager`'s constructor runs at module load (before any
gesture) and kicks off 13 `fetch()` calls, but these retrieve **ArrayBuffers only** — no
`AudioContext` is created and nothing is decoded or played, so autoplay policy is not
tripped (AC6) and there is no audio thread work at startup. Decode happens once, lazily, in
`unlock()`. The `music-loop.mp3` is kept low-bitrate/mono and short (~16s) so the largest
fetch stays small; combined, startup network/CPU cost is bounded and does not regress first
paint (AC7).
